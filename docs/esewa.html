How eSewa Works

ℹ️

eSewa uses a form-submission model. Your backend generates a signed payload. Your frontend submits it as an HTML form directly to eSewa.

Your Backend: buildFormPayload() → HMAC-SHA256 signed payload → return to frontend │ Frontend: POST form fields to payload.formActionUrl() │ User pays on eSewa payment page │ eSewa redirects to: yourSuccessUrl?data=BASE64_JSON │ ⚠️ DO NOT trust this redirect alone! │ Your Backend: verifyCallback(data) Step 1 — decode Base64 → EsewaCallbackData Step 2 — re-compute HMAC signature, compare Step 3 — call eSewa status API for confirmation │ ✅ mark order as paid only when isPaymentSuccessful() = true

Key differences from Khalti

Feature Khalti eSewa
Flow API-first Form POST
Signature API Key header HMAC-SHA256
Amount unit Paisa NPR directly
Verify auth Bearer key No auth on status API
Registration Self-service sandbox Self-service sandbox

buildFormPayload()

Simple — amount and UUID only

// Generate UUID — MUST save to DB before returning!
String uuid = EsewaClient.generateTransactionUuid();
orderRepo.saveUuid(orderId, uuid);

// ⚠️ Amount is NPR directly — NOT paisa like Khalti
EsewaFormPayload payload = esewaClient.buildFormPayload(
    new BigDecimal("100.00"),
    uuid
);

// Return payload to frontend — it will POST the form

With tax and charges

EsewaFormPayload payload = esewaClient.buildFormPayload(
    new BigDecimal("100.00"),  // base amount
    new BigDecimal("13.00"),   // tax
    uuid,
    BigDecimal.ZERO,           // service charge
    BigDecimal.ZERO            // delivery charge
    // total = 100 + 13 = 113.00
);
⚠️

eSewa uses NPR directly — not paisa. NPR 100 → send new BigDecimal("100.00"). This is the opposite of Khalti which uses paisa.

Payload fields returned

Field Type Description
formActionUrl() String The eSewa URL to POST the form to
amount() String Base amount in NPR
taxAmount() String Tax amount in NPR
totalAmount() String Sum of all amounts — used in signature
transactionUuid() String Your unique transaction ID
productCode() String Your eSewa merchant/product code
signature() String HMAC-SHA256 Base64 signature
signedFieldNames() String Always total_amount,transaction_uuid,product_code

Frontend Form Submission

The frontend must POST all payload fields as hidden inputs to payload.formActionUrl(). Here is an Angular / TypeScript example: 

// Angular / TypeScript
initiateEsewa(payload: EsewaFormPayload): void {
    const form = document.createElement('form');
    form.method = 'POST';
    form.action = payload.form_action_url;

    const fields = [
        'amount', 'tax_amount', 'total_amount',
        'transaction_uuid', 'product_code',
        'product_service_charge', 'product_delivery_charge',
        'success_url', 'failure_url',
        'signed_field_names', 'signature'
    ];

    fields.forEach(field => {
        const input = document.createElement('input');
        input.type  = 'hidden';
        input.name  = field;
        input.value = (payload as any)[field] ?? '';
        form.appendChild(input);
    });

    document.body.appendChild(form);
    form.submit();
}
ℹ️

The form is submitted directly from the browser to eSewa. Your backend is not involved in this step. After the user pays, eSewa redirects back to your success_url with a data query parameter.

verifyCallback()

eSewa redirects to your successUrl with ?data=BASE64_JSON. Call verifyCallback(data) which automatically:

  • Decodes the Base64 data
  • Re-computes the HMAC-SHA256 signature and compares
  • Calls the eSewa status API for final confirmation
// eSewa redirects to: yourSuccessUrl?data=BASE64_JSON
@GetMapping("/esewa/callback")
public ResponseEntity<String> callback(@RequestParam String data) {

    // verifyCallback does 3 things automatically:
    // 1. Decode Base64 → EsewaCallbackData
    // 2. Verify HMAC-SHA256 signature (tamper protection)
    // 3. Call eSewa status API for final confirmation
    EsewaClient.EsewaVerificationResult result =
        esewaClient.verifyCallback(data);

    if (!result.isPaymentSuccessful()) {
        return ResponseEntity.badRequest()
            .body("Not confirmed: " + result.statusResponse().status());
    }

    // Safe to mark order as paid
    String uuid  = result.callbackData().transactionUuid();
    String refId = result.statusResponse().refId();
    orderRepo.markPaid(uuid, refId);

    return ResponseEntity.ok("Payment confirmed");
}

EsewaVerificationResult fields

Method Returns Description
isPaymentSuccessful() boolean true only when signature verified AND status is COMPLETE
callbackData() EsewaCallbackData Decoded Base64 callback — contains transactionUuid, status
statusResponse() EsewaStatusResponse Response from eSewa status API — contains refId
🔴

Never trust the redirect URL alone. Always call verifyCallback(data). It verifies the HMAC signature — a mismatch throws EsewaException with the message signature verification FAILED.

checkStatus()

Poll the eSewa status API directly — useful for background verification jobs or when you want to check status without a callback. 

EsewaStatusResponse status = esewaClient.checkStatus(
    "your-transaction-uuid",
    "100.00"  // exact total amount from original transaction
);

if (status.isPaymentSuccessful()) {
    orderRepo.markPaid(uuid);
}

Status values

Status string isPaymentSuccessful() Meaning
COMPLETE true Payment fully completed — safe to fulfil order
INCOMPLETE false Payment not completed
other / null false Unknown state — do NOT fulfil order
⚠️

eSewa uses COMPLETE — not COMPLETED like Khalti. This is a common source of bugs when switching between gateways.

Security Rules

🔴

Never trust the redirect URL alone. Always call verifyCallback() which verifies the HMAC-SHA256 signature before calling the status API.

✅

Save transactionUuid to your database before returning the form payload to the frontend.

✅

The reactive client uses MessageDigest.isEqual() for constant-time HMAC comparison — prevents timing attacks. 

// ✅ Correct flow
String uuid = EsewaClient.generateTransactionUuid();
orderRepo.saveUuid(orderId, uuid);              // SAVE FIRST
EsewaFormPayload payload =
    esewaClient.buildFormPayload(amount, uuid); // THEN build
return payload;                                 // return to frontend

// ✅ Correct verification — 3 steps happen automatically
EsewaClient.EsewaVerificationResult result =
    esewaClient.verifyCallback(data);
// Step 1: Base64 decode
// Step 2: HMAC verify — throws EsewaException on mismatch
// Step 3: Status API call

// ❌ NEVER do this
if (request.getParam("status").equals("COMPLETE")) {
    orderRepo.markPaid(); // DANGEROUS — no HMAC check
}

Reactive Usage

Use EsewaReactiveClient from nepal-pay-spring-boot-reactive-starter. buildFormPayload() is synchronous — only verifyCallback() and checkStatus() return Mono.

Build form payload — synchronous

// No Mono needed — synchronous, no HTTP call
public EsewaFormPayload startEsewa(BigDecimal amount) {
    String uuid = EsewaReactiveClient.generateTransactionUuid();
    orderRepo.saveUuid(orderId, uuid);  // save first!
    return esewaReactiveClient.buildFormPayload(amount, uuid);
}

Verify callback — reactive

@GetMapping("/esewa/callback")
public Mono<ResponseEntity<String>> callback(
        @RequestParam String data) {

    return esewaReactiveClient.verifyCallback(data)
        .flatMap(result -> {
            if (!result.isPaymentSuccessful()) {
                return Mono.just(ResponseEntity.badRequest()
                    .body("Not confirmed"));
            }
            String uuid = result.callbackData().transactionUuid();
            return orderRepo.markPaid(uuid)
                .thenReturn(ResponseEntity.ok("Payment confirmed"));
        });
}

Check status — reactive

public Mono<Boolean> checkEsewa(String uuid, String amount) {
    return esewaReactiveClient
        .checkStatus(uuid, amount)
        .map(EsewaStatusResponse::isPaymentSuccessful);
}
ℹ️

The reactive verifyCallback() uses MessageDigest.isEqual() for constant-time HMAC comparison — prevents timing attacks. All errors are emitted as Mono.error() signals, never thrown.
