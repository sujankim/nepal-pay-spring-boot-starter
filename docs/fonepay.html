How Fonepay Works

ℹ️

Fonepay uses a URL redirect model. Your backend builds a signed URL. The frontend navigates the user directly to it. No HTTP API calls are made from your backend.

Your Backend: buildRedirectParams() → HMAC-SHA512 signature (hex lowercase) → returns FonepayRedirectParams with full redirectUrl ↓ Frontend: window.location.href = params.redirectUrl() ↓ User pays on Fonepay (QR, bank, wallet) ↓ Fonepay redirects to your returnUrl: ?PRN=xxx&PID=xxx&PS=success&RC=200&DV=HMAC_HEX ↓ ⚠️ NEVER trust PS=success alone — DV can be faked! ↓ Your Backend: verifyCallback(callback) → re-compute HMAC-SHA512, compare with DV → check PS = "success" ↓ ✅ mark as paid only when both match

Key Differences from Other Gateways

Feature Khalti eSewa Fonepay
Flow API POST Form POST URL redirect (GET)
Signature API key header HMAC-SHA256 Base64 HMAC-SHA512 hex
Amount unit Paisa (long) NPR (BigDecimal) NPR (double)
Server HTTP calls Yes — RestClient Yes — RestClient None
Retry support Yes Yes N/A
Reactive KhaltiReactiveClient EsewaReactiveClient Use blocking FonepayClient
ℹ️

Because Fonepay makes no server-side HTTP calls, retry does not apply and the blocking FonepayClient works perfectly inside reactive WebFlux apps — no wrapping needed.

buildRedirectParams()

// PRN must be 3–25 chars, unique per transaction
String prn = "FP-" + orderId;
orderRepo.savePrn(orderId, prn); // save BEFORE returning!

FonepayRedirectParams params = fonepayClient.buildRedirectParams(
    FonepayPaymentRequest.builder()
        .prn(prn)
        .amount(100.0)        // ⚠️ NPR directly — not paisa!
        .remarks1("Pro Plan") // shown to user
        .remarks2("MyApp")    // optional, max 50 chars
        .build()
);

// Return params.redirectUrl() to frontend
⚠️

Amount is NPR as a double — not paisa. NPR 100 → send 100.0. PRN must be 3–25 characters.

Request builder fields

Method Required Description
.prn(String) Required Product Reference Number — unique, 3–25 chars, store in DB
.amount(double) Required Amount in NPR as a double — not paisa
.remarks1(String) Required Payment description shown to user, max 160 chars
.remarks2(String) Optional Additional remarks, max 50 chars

FonepayRedirectParams fields

Method Description
redirectUrl() Full signed URL — redirect the user to this
prn() Your Product Reference Number
dv() HMAC-SHA512 signature (hex lowercase)
amt() Amount string
ru() Return URL

Frontend Redirect

// Angular
initiatePayment(orderId: string, amount: number): void {
    this.paymentService.startFonepay(orderId, amount).subscribe(res => {
        window.location.href = res.redirect_url;
    });
}

// Plain JavaScript
fetch('/api/payment/fonepay/initiate', { method: 'POST', ... })
  .then(res => res.json())
  .then(data => window.location.href = data.redirect_url);
ℹ️

Unlike eSewa and ConnectIPS, Fonepay uses a simple window.location.href redirect — no form submission required. The entire signed URL is built server-side and returned to the frontend.

verifyCallback()

Fonepay redirects to your returnUrl with all payment parameters as GET query parameters. Always verify the DV signature before trusting PS=success. 

@GetMapping("/fonepay/callback")
public ResponseEntity<String> callback(
        @RequestParam String PRN,
        @RequestParam String PID,
        @RequestParam String PS,
        @RequestParam String RC,
        @RequestParam String UID,
        @RequestParam String BC,
        @RequestParam String INI,
        @RequestParam(name = "P_AMT") String pAmt,
        @RequestParam(name = "R_AMT") String rAmt,
        @RequestParam(name = "DV")    String dv) {

    FonepayCallbackResponse callback =
        FonepayCallbackResponse.of(
            PRN, PID, PS, RC, UID, BC, INI, pAmt, rAmt, dv);

    // ✅ ALWAYS verify signature first — redirect params can be faked!
    FonepayClient.FonepayVerificationResult result =
        fonepayClient.verifyCallback(callback);

    if (!result.isPaymentSuccessful()) {
        return ResponseEntity.ok(
            Map.of("verified", false, "message", "Payment not successful"));
    }

    orderService.markAsPaid(PRN);
    return ResponseEntity.ok(
        Map.of("verified", true, "message", "Payment confirmed"));
}

FonepayVerificationResult

Method Returns Description
isPaymentSuccessful() boolean true only when HMAC matched AND PS=success
verified() boolean Same as isPaymentSuccessful()
callbackResponse() FonepayCallbackResponse The original callback parameters
⚠️

verifyCallback() throws FonepayException when the HMAC-SHA512 signature does not match. Treat this as a potential fraud attempt and log it. Never call callback.paymentStatus() directly — it bypasses HMAC verification.

Security Rules

🔴

Never trust PS=success alone. The DV signature is the only real proof. Always call verifyCallback() which re-computes HMAC-SHA512.

✅

Save the PRN to your database before returning the redirect URL to the frontend.

✅

verifyCallback() throws FonepayException on signature mismatch — treat this as a potential fraud attempt and log it. 

// ✅ Correct flow
String prn = "FP-" + orderId;
orderRepo.savePrn(orderId, prn);        // SAVE FIRST
FonepayRedirectParams params =
    fonepayClient.buildRedirectParams(request);
return params.redirectUrl();            // return to frontend

// ✅ Correct verification
FonepayClient.FonepayVerificationResult result =
    fonepayClient.verifyCallback(callback);
// Throws FonepayException if DV mismatch

if (result.isPaymentSuccessful()) {
    orderRepo.markPaid(PRN);
}

// ❌ NEVER do this
if ("success".equals(ps)) {
    orderRepo.markPaid(PRN); // DANGEROUS — DV not checked
}

Reactive Apps — Use Blocking Client

Fonepay makes zero server-side HTTP calls. The blocking FonepayClient is safe to use inside WebFlux applications — it never touches the event loop.

Add alongside the reactive starter

<!-- Reactive starter for Khalti, eSewa, ConnectIPS -->
<dependency>
    <groupId>io.github.sujankim</groupId>
    <artifactId>nepal-pay-spring-boot-reactive-starter</artifactId>
    <version>1.2.0</version>
</dependency>

<!-- Blocking starter for FonepayClient only -->
<dependency>
    <groupId>io.github.sujankim</groupId>
    <artifactId>nepal-pay-spring-boot-3-starter</artifactId>
    <version>1.2.0</version>
</dependency>

Wrap in Mono.fromCallable if needed

@Service
@RequiredArgsConstructor
public class FonepayService {

    // Blocking client — works fine in reactive apps
    private final FonepayClient fonepayClient;
    private final OrderRepository orderRepo;   // ← declare injected repo

    public String buildRedirect(String orderId, double amountNPR) {
        String prn = "FP-" + orderId;
        orderRepo.savePrn(orderId, prn);

        return fonepayClient.buildRedirectParams(
            FonepayPaymentRequest.builder()
                .prn(prn)
                .amount(amountNPR)
                .remarks1("Pro Plan")
                .build()
        ).redirectUrl();
    }

    // Optional: wrap in Mono if needed for reactive pipeline
    public Mono<Boolean> verifyFonepay(
            FonepayCallbackResponse callback) {
        return Mono.fromCallable(() ->
            fonepayClient.verifyCallback(callback)
                         .isPaymentSuccessful());
    }
}

Callback Parameters

Fonepay redirects to your returnUrl with these GET query parameters:

Param Description Example
PRN Your original Product Reference Number FP-ORD-001
PID Fonepay merchant code TEST_MERCHANT
PS Payment status success or failed
RC Response code 200
UID Fonepay unique transaction ID uid-001
BC Bank code GBIME
INI Transaction initiator 9800000001
P_AMT Paid amount in NPR 100
R_AMT Refund amount 0
DV HMAC-SHA512 signature (uppercase hex) — verify this! A1B2C3...

Signature algorithm — how DV is computed

// Field order is MANDATORY per Fonepay spec — request signature:
// PID,MD,PRN,AMT,CRN,DT,R1,R2,RU

// Response verification field order:
// PRN,PID,PS,RC,UID,BC,INI,P_AMT,R_AMT

// Hash: HMAC-SHA512(secretKey, message, UTF-8) → hex lowercase
// DV from Fonepay is UPPERCASE hex — NepalPay normalizes before comparing
