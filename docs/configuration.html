Khalti Properties

KhaltiClient (and KhaltiReactiveClient) is created when nepalpay.khalti.secret-key is present.

Property Required Default Description
secret-key ✅ — Khalti secret key. Creates bean when present.
return-url ✅ — URL Khalti redirects to after payment.
website-url ✅ — Your merchant website URL.
sandbox — true true = dev.khalti.com · false = khalti.com
timeout-seconds — 10 HTTP connect and read timeout in seconds.
retry.* — disabled See Retry Config →

eSewa Properties

EsewaClient (and EsewaReactiveClient) is created when nepalpay.esewa.secret-key is present.

Property Required Default Description
secret-key ✅ — HMAC-SHA256 secret key. Sandbox: 8gBm/:&EnhH.1/q
product-code ✅ — Sandbox: EPAYTEST · Production: your merchant code
success-url ✅ — Redirect on successful payment with ?data=BASE64
failure-url ✅ — Redirect on failed or canceled payment
sandbox — true true = rc-epay.esewa.com.np · false = epay.esewa.com.np
timeout-seconds — 10 HTTP timeout for status API calls in seconds
retry.* — disabled Applies to checkStatus() inside verifyCallback()

ConnectIPS Properties

⚠️

Requires NCHL merchant registration before use. Contact connectips@nchl.com.np .

Property Required Default Description
merchant-id ✅ — NCHL integer merchant ID. Creates bean when present.
app-id ✅ — Application ID from NCHL.
app-name ✅ — Application name from NCHL.
app-password ✅ — Used for HTTP Basic Auth on the validate API.
pfx-path ✅ — Spring Resource path. file:/app/CREDITOR.pfx or classpath:CREDITOR.pfx
pfx-password ✅ — Password for the CREDITOR.pfx file.
sandbox — true true = uat.connectips.com · false = connectips.com
timeout-seconds — 30 HTTP timeout in seconds. Default 30s — intentionally longer because ConnectIPS validates via NCHL to bank systems. Increase for slower banks.
retry.* — disabled Applies to validateTransaction()

Fonepay Properties

ℹ️

No retry or timeout for Fonepay. Fonepay uses URL redirect and makes zero server-to-server HTTP calls.

Property Required Default Description
merchant-code ✅ — Your Fonepay PID (merchant code).
secret-key ✅ — HMAC-SHA512 secret key. Creates bean when present.
return-url ✅ — Fonepay redirects here with all callback params after payment.
sandbox — true true = dev.fonepay.com · false = fonepay.com

Retry Configuration

ℹ️

Retry is disabled by default. You must explicitly set enabled: true per gateway. The blocking starter uses Thread.sleep(). The reactive starter uses Reactor's Retry.backoff() — fully non-blocking. 

nepalpay:
  khalti:
    retry:
      enabled: true           # opt-in — false by default
      max-attempts: 3         # retries after first failure (4 total calls)
      initial-delay-ms: 500   # wait 500ms before first retry
      multiplier: 2.0         # 500 → 1000 → 2000ms
      max-delay-ms: 5000      # never wait more than 5 seconds

What gets retried

Scenario Retried? Reason
5xx server errors ✅ Yes Gateway temporarily down — may recover
Network timeout / connection reset ✅ Yes Transient network issue
400 Bad Request ❌ No Invalid request — retrying won't help
401 Unauthorized ❌ No Wrong key — retrying won't help
Signature failure ❌ No Security alert — must never retry

Retry Properties Reference

Property Default Description
retry.enabled false Must be true to activate.
retry.max-attempts 3 Retries after first failure. Total calls = max-attempts + 1.
retry.initial-delay-ms 500 Milliseconds before first retry.
retry.multiplier 2.0 Delay multiplier per retry. Use 1.0 for constant delay. Must be >= 1.0.
retry.max-delay-ms 5000 Maximum wait between retries in milliseconds.

Exponential backoff with jitter

# initial-delay-ms=500, multiplier=2.0, max-delay-ms=5000, max-attempts=3

First call  → fails 5xx → wait ~500ms  (±10% jitter)
Retry 1     → fails 5xx → wait ~1000ms (500 × 2.0)
Retry 2     → fails 5xx → wait ~2000ms (1000 × 2.0)
Retry 3     → fails 5xx → throw exception — no more retries

# Jitter: ±10% random offset prevents thundering herd
# All clients retry on 4xx → never

nepalpay:

ℹ️

Health indicators are config-only — no outbound HTTP ping to the gateway APIs. This avoids false DOWN states caused by sandbox rate limits during startup health checks. Real payment failures are tracked via Micrometer counters.

Production Configuration

🔴

Never hardcode production keys. Always use environment variables. Never commit real credentials to Git. 

nepalpay:

  khalti:
    secret-key:  ${KHALTI_SECRET_KEY}
    return-url:  https://yourapp.com/api/khalti/callback
    website-url: https://yourapp.com
    sandbox: false                      # ← production

  esewa:
    secret-key:   ${ESEWA_SECRET_KEY}
    product-code: ${ESEWA_PRODUCT_CODE} # ← your real merchant code
    success-url:  https://yourapp.com/api/esewa/callback
    failure-url:  https://yourapp.com/payment/failed
    sandbox: false                      # ← production

  connectips:
    merchant-id:     ${CONNECTIPS_MERCHANT_ID}

    app-name:        ${CONNECTIPS_APP_NAME}

Override Beans

Every NepalPay bean uses @ConditionalOnMissingBean. Define your own bean and NepalPay steps aside automatically. 

@Configuration
public class MyPaymentConfig {

    // Override KhaltiClient — NepalPay's auto-configured bean is skipped
    @Bean
    public KhaltiClient customKhaltiClient(
            NepalPayProperties props,
            RestClient.Builder builder) {
        return new KhaltiClient(props.khalti(), builder);
    }
}

Works for all four clients: KhaltiClient, EsewaClient, ConnectIpsClient, FonepayClient — and their reactive equivalents.
