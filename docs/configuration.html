ConnectIPS Properties

⚠️

Requires NCHL merchant registration. Contact connectips@nchl.com.np.

PropertyRequiredDefaultDescription
merchant-id✅—NCHL integer merchant ID. Creates bean when present.
app-id✅—Application ID from NCHL.
app-name✅—Application name from NCHL.
app-password✅—Used for HTTP Basic Auth on validate API.
pfx-path✅—Path to CREDITOR.pfx. Example: file:/app/CREDITOR.pfx
pfx-password✅—Password for CREDITOR.pfx file.
sandbox—truetrue = uat.connectips.com · false = production
timeout-seconds—30HTTP timeout in seconds. Default is 30s — intentionally longer than other gateways because ConnectIPS validates via NCHL to bank systems which can be slower.
retry.*—disabledApplies to validateTransaction().
ℹ️

ConnectIPS timeout defaults to 30 seconds — higher than Khalti and eSewa's 10s. Bank systems accessed via NCHL can legitimately take longer to respond. Increase this value if you observe timeouts with slower banks.

Retry Configuration

ℹ️

Retry is disabled by default. You must explicitly set enabled: true to activate it per gateway.

Example

nepalpay:
  khalti:
    secret-key: ${KHALTI_SECRET_KEY}
    return-url:  ${KHALTI_RETURN_URL}
    sandbox: true
    retry:
      enabled: true           # opt-in — false by default
      max-attempts: 3         # retries after first failure (4 total calls max)
      initial-delay-ms: 500   # wait 500ms before first retry
      multiplier: 2.0         # double delay each retry: 500 → 1000 → 2000
      max-delay-ms: 5000      # never wait more than 5 seconds

What gets retried

ScenarioRetried?Reason
5xx server errors✅ YesGateway temporarily down — may recover
Network timeout / connection reset✅ YesTransient network issue
400 Bad Request❌ NoInvalid request — retrying won't help
401 Unauthorized❌ NoWrong key — retrying won't help
Signature failure❌ NoSecurity alert — must not retry

Which gateways support retry

GatewayMethodsRetry?
KhaltiinitiatePayment(), lookupPayment(), refundPayment()✅ Yes
eSewacheckStatus() (inside verifyCallback())✅ Yes
ConnectIPSvalidateTransaction()✅ Yes
FonepayAll methods❌ N/A

Retry Properties Reference

PropertyDefaultDescription
retry.enabledfalseMust be true to activate. Disabled by default.
retry.max-attempts3Retries after first failure. Total calls = max-attempts + 1.
retry.initial-delay-ms500Milliseconds before first retry.
retry.multiplier2.0Delay multiplier per retry. Use 1.0 for constant delay. Must be >= 1.0.
retry.max-delay-ms5000Maximum wait between retries in milliseconds.

How exponential backoff works

# initial-delay-ms=500, multiplier=2.0, max-delay-ms=5000, max-attempts=3

First call  → fails with 500 → wait 500ms (±jitter)
Retry 1     → fails with 500 → wait 1000ms (500 × 2.0)
Retry 2     → fails with 500 → wait 2000ms (1000 × 2.0)
Retry 3     → fails with 500 → no more retries → throw exception

# Jitter: ±10% random offset added to every wait
# Prevents all users retrying at the exact same millisecond

Production Configuration

🔴

Never hardcode production keys. Always use environment variables. Never commit real credentials to Git.

nepalpay:

  khalti:
    secret-key:  ${KHALTI_SECRET_KEY}
    return-url:  https://yourapp.com/api/khalti/callback
    website-url: https://yourapp.com
    sandbox: false

  esewa:
    secret-key:   ${ESEWA_SECRET_KEY}
    product-code: ${ESEWA_PRODUCT_CODE}
    success-url:  https://yourapp.com/api/esewa/callback
    failure-url:  https://yourapp.com/payment/failed

  connectips:

Override Beans

Every NepalPay bean uses @ConditionalOnMissingBean. Define your own bean and NepalPay steps aside automatically.

@Configuration
public class MyPaymentConfig {

    @Bean
    public KhaltiClient customKhaltiClient(
            NepalPayProperties props,
            RestClient.Builder builder) {
        // your custom configuration here
        return new KhaltiClient(props.khalti(), builder);
    }
}

Works for all four clients: KhaltiClient, EsewaClient, ConnectIpsClient, FonepayClient.
