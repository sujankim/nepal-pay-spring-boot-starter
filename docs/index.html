Why NepalPay

Stop writing payment boilerplate

Every project needs payments. None of them should need 500 lines of HMAC signing code — or 200 lines of Reactor pipelines.

⚡

5-Minute Setup

Add dependency → configure keys → inject and use. No @Bean, no config class, no boilerplate at all. Works the same in blocking and reactive apps.

🔄

Reactive (WebFlux)

New in v1.1.0 — Mono<T> responses via WebClient. Fully non-blocking for WebFlux apps. Same YAML config as blocking.

🔐

Secure by Default

HMAC-SHA256, HMAC-SHA512, RSA-SHA256, server-side verification, constant-time HMAC comparison — all implemented correctly and tested.

🔁

Retry Built In

Exponential backoff with jitter. Blocking uses Thread.sleep(). Reactive uses Reactor's Retry.backoff(). Opt-in, configurable per gateway.

💳

Refund Support

Khalti full and partial refunds supported. More gateways coming with community contributions. Blocking and reactive.

🧪

Fully Tested

MockWebServer HTTP tests, StepVerifier reactive tests, signature correctness, retry behavior. 400+ tests total.

✅ With NepalPay 
// Inject — auto-configured, zero setup
@Service
@RequiredArgsConstructor
public class PaymentService {

    private final KhaltiClient khaltiClient;

    public String initiate(
            String orderId, long nprAmount) {

        var res = khaltiClient.initiatePayment(
            KhaltiInitiateRequest.builder()
                .amount(nprAmount * 100L) // NPR → paisa
                .purchaseOrderId(orderId)
                .purchaseOrderName("Pro Plan")
                .build()
        );

        // Save res.pidx() to DB, redirect to:
        return res.paymentUrl();
    }
}

New in v1.2.0

Micrometer metrics + health indicators

Add spring-boot-starter-actuator — NepalPay automatically records timers, counters, and registers a health indicator per gateway.

⏱

Operation Timers

Every HTTP call timed and tagged by gateway, operation, and status. Use for latency dashboards in Grafana.

🔢

Retry Counters

Each retry attempt increments a counter per operation. Shows how often gateway 5xx errors force your app to retry.

🛡

Signature Failure Counter

nepalpay.esewa.callback.signature.failed increments on every HMAC mismatch. Alert on a spike — it means tampering.

🩺

Health per Gateway

Each configured gateway appears as its own component under /actuator/health. Config-only — no live pings.

⚡

Reactive-Safe Timing

Uses Timer.Sample with doOnSuccess/doOnError on the Mono. Fully non-blocking throughout.

🔘

Opt-Out Anytime

Set nepalpay.metrics.enabled=false or nepalpay.health.enabled=false. No Actuator on classpath means nothing registers.

Retry with Exponential Backoff

Built-in retry — opt-in per gateway

Retry is disabled by default. Enable it for any HTTP-backed gateway with four YAML lines. 

nepalpay:
  khalti:
    retry:
      enabled: true
      max-attempts: 3
      initial-delay-ms: 500
      multiplier: 2.0
      max-delay-ms: 5000
Gateway Retry Applies To Retry?
Khalti initiatePayment(), lookupPayment(), refundPayment() ✅ Yes
eSewa checkStatus() inside verifyCallback() ✅ Yes
ConnectIPS validateTransaction() ✅ Yes
Fonepay URL redirect — no HTTP calls ❌ N/A

Retries: 5xx errors and network timeouts  ·  No retries: 4xx errors and signature failures  ·  Note: Refund retries require idempotency — verify the refund status with lookupPayment() before retrying to avoid double-refunds.

Security

Never trust redirect parameters alone

Payment gateways redirect users back with URL parameters. These can be faked. Always verify server-side.

✅

Server-side Only

Secret keys never leave your backend. lookupPayment() and verifyCallback() enforce server-side confirmation.

🔏

HMAC Verification

eSewa and Fonepay callback signatures are always verified before marking any payment as successful. Constant-time comparison prevents timing attacks.

💰

Amount Validation

isAmountValid(expectedPaisa) on Khalti lookup detects tampered amounts. Always call this after success.

❌

Never trust redirect URL parameters alone — ?status=Completed, ?PS=success, ?data=BASE64 can all be faked.

❌

Never put secret keys in frontend code. Never hardcode credentials in source. Use environment variables for all real credentials.

❌

Never commit CREDITOR.pfx to Git. Add *.pfx to .gitignore immediately.
