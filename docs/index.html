Why NepalPay

Stop writing payment boilerplate

Every project needs payments. None of them should need 500 lines of HMAC signing code.

⚡

5-Minute Setup

Add dependency → configure keys → inject and use. No @Bean, no config class, no boilerplate at all.

🔐

Secure by Default

HMAC-SHA256, HMAC-SHA512, RSA-SHA256, server-side verification — all implemented correctly and tested.

🔁

Retry Built In

Exponential backoff with jitter for all HTTP-calling gateways. Opt-in, configurable per gateway.

💳

Refund Support

Khalti full and partial refunds supported. More gateways coming with community contributions.

🧪

Fully Tested

MockWebServer HTTP tests, signature correctness tests, retry behavior tests. 80+ tests total.

🔀

Boot 3 and Boot 4

Separate modules — identical public API. Switch starters without changing any application code.

✅ With NepalPay 
// Inject — auto-configured, zero setup
@Service
@RequiredArgsConstructor
public class PaymentService {

    private final KhaltiClient khaltiClient;

    public String initiate(String orderId, long nprAmount) {

        var res = khaltiClient.initiatePayment(
            KhaltiInitiateRequest.builder()
                .amount(nprAmount * 100L) // NPR → paisa
                .purchaseOrderId(orderId)
                .purchaseOrderName("Pro Plan")
                .build()
        );

        // Save res.pidx() to DB, redirect to:
        return res.paymentUrl();
    }
}
