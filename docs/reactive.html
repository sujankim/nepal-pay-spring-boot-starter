Blocking vs Reactive

Why Reactive?

Spring WebFlux applications run on a non-blocking event loop (Netty). Calling a blocking HTTP client (like RestClient) inside a WebFlux handler blocks the event loop thread — killing the performance benefit of reactive.

❌ Blocking client in WebFlux 
// BAD — blocks the event loop thread
@GetMapping("/pay")
public Mono<String> pay() {
    // khaltiClient uses RestClient internally
    // This blocks the Netty thread!
    KhaltiInitiateResponse res =
        khaltiClient.initiatePayment(req);
    return Mono.just(res.paymentUrl());
}
✅ Reactive client in WebFlux 
// GOOD — fully non-blocking pipeline
@GetMapping("/pay")
public Mono<String> pay() {
    // khaltiReactiveClient uses WebClient
    // Non-blocking all the way through
    return khaltiReactiveClient
        .initiatePayment(req)
        .map(KhaltiInitiateResponse::paymentUrl);
}
ℹ️

Using Spring MVC (not WebFlux)? Stick with the blocking starter — nepal-pay-spring-boot-3-starter or nepal-pay-spring-boot-4-starter. There is no benefit to reactive clients in a blocking MVC application.

How It Works

Concern Blocking Starter Reactive Starter

Retry while loop + Thread.sleep() Retry.backoff()

Reactive contract rule

All synchronous work that can throw — input validation, RSA signing, HMAC computation — is wrapped inside Mono.defer() so any exception is emitted as a reactive error signal, never thrown outside the pipeline. 

// ✅ Correct — ConnectIPS validateTransaction() example
public Mono<ConnectIpsValidateResponse> validateTransaction(
        String txnId, String referenceId, long txnAmtPaisa) {

    return Mono.defer(() -> {
        // Validation inside defer — becomes Mono.error if invalid
        try {
            validatePaymentInput(txnId, referenceId, txnAmtPaisa);
        } catch (ConnectIpsException e) {
            return Mono.error(e);
        }

        // RSA signing inside defer — becomes Mono.error if fails
        String token;
        try {
            token = generateRsaToken(buildMessage(...));
        } catch (ConnectIpsException e) {
            return Mono.error(e);
        }

        return executeValidateRequest(referenceId, txnAmtPaisa, token);
    });
}

Khalti — Reactive

Your Backend: khaltiReactiveClient.initiatePayment() → WebClient POST to Khalti API (non-blocking) → Mono<KhaltiInitiateResponse> ↓ doOnNext: save pidx to DB ↓ map: return paymentUrl to frontend ↓ Frontend redirects user to paymentUrl ↓ User pays on Khalti ↓ Khalti redirects to your returnUrl ↓ ⚠️ NEVER trust redirect params alone! ↓ khaltiReactiveClient.lookupPayment(pidx) → Mono<KhaltiLookupResponse> ↓ filter: isPaymentSuccessful() ↓ flatMap: mark order as paid

Initiate payment

public Mono<String> startKhalti(String orderId, long amountNPR) {
    return khaltiReactiveClient.initiatePayment(
        KhaltiInitiateRequest.builder()
            .amount(amountNPR * 100L)      // NPR → paisa
            .purchaseOrderId(orderId)
            .purchaseOrderName("Pro Plan")
            .build()
    )
    // IMPORTANT: save pidx BEFORE returning paymentUrl
    .doOnNext(res -> orderRepo.savePidx(orderId, res.pidx()))
    .map(KhaltiInitiateResponse::paymentUrl);
}

Verify after callback

// Called at your returnUrl
@GetMapping("/khalti/callback")
public Mono<ResponseEntity<String>> callback(
        @RequestParam String pidx) {

    return khaltiReactiveClient.lookupPayment(pidx)
        .flatMap(lookup -> {
            if (!lookup.isPaymentSuccessful()) {
                return Mono.just(ResponseEntity.badRequest()
                    .body("Not confirmed: " + lookup.status()));
            }
            return orderRepo.markPaid(pidx, lookup.transactionId())
                .thenReturn(ResponseEntity.ok("Payment confirmed"));
        });
}

Full refund

public Mono<Boolean> refundKhalti(String transactionId) {

public Mono<Boolean> partialRefund(

eSewa — Reactive

ℹ️

buildFormPayload() is synchronous — it generates an HMAC-SHA256 signature locally and returns immediately. No HTTP call, no Mono needed. Only checkStatus() and verifyCallback() are reactive.

Build form payload — synchronous

// No Mono needed — synchronous, no HTTP call
public EsewaFormPayload startEsewa(BigDecimal amountNPR) {
    String uuid = EsewaReactiveClient.generateTransactionUuid();

    // IMPORTANT: save uuid to DB BEFORE returning!
    orderRepo.saveUuid(orderId, uuid);

    // ⚠️ Amount is NPR directly — NOT paisa like Khalti
    return esewaReactiveClient.buildFormPayload(amountNPR, uuid);
    // Frontend POSTs all fields to payload.formActionUrl()
}

Verify callback — reactive

@GetMapping("/esewa/callback")
public Mono<ResponseEntity<String>> callback(
        @RequestParam String data) {

    return esewaReactiveClient.verifyCallback(data)
        .flatMap(result -> {
            if (!result.isPaymentSuccessful()) {
                return Mono.just(ResponseEntity.badRequest()
                    .body("Not confirmed: "
                        + result.statusResponse().status()));
            }
            String uuid = result.callbackData().transactionUuid();
            return orderRepo.markPaid(uuid)
                .thenReturn(ResponseEntity.ok("Payment confirmed"));
        });
}

Check status directly — reactive

public Mono<Boolean> checkEsewa(String uuid, String amount) {
    return esewaReactiveClient
        .checkStatus(uuid, amount)
        .map(EsewaStatusResponse::isPaymentSuccessful);
}

ConnectIPS — Reactive

ℹ️

buildFormPayload() is synchronous — RSA signing happens locally, no HTTP call. Only validateTransaction() is reactive. Both validation and RSA signing inside Mono.defer() — all errors are reactive signals.

Build form payload — synchronous

public ConnectIpsFormPayload startConnectIps(
        String orderId, long amountNPR) {

    String txnId = "TXN-" + orderId + "-"
            + System.currentTimeMillis();

    // IMPORTANT: save txnId to DB BEFORE returning!
    orderRepo.saveTxnId(orderId, txnId);

    // Synchronous — RSA signing, no HTTP, no Mono
    return connectIpsReactiveClient.buildFormPayload(
        ConnectIpsPaymentRequest.builder()
            .txnId(txnId)
            .amountNPR(amountNPR)     // auto-converts to paisa
            .referenceId(orderId)
            .build()
    );
}

Validate transaction — reactive

@GetMapping("/connectips/callback")
public Mono<ResponseEntity<String>> callback(
        @RequestParam String txnId,
        @RequestParam String referenceId,
        @RequestParam long txnAmt) {

    return connectIpsReactiveClient
        .validateTransaction(txnId, referenceId, txnAmt)
        .flatMap(res -> {
            if (!res.isPaymentSuccessful()) {
                return Mono.just(ResponseEntity.badRequest()
                    .body("Not confirmed: " + res.statusDesc()));
            }
            return orderRepo.markPaid(referenceId)
                .thenReturn(ResponseEntity.ok("Payment confirmed"));
        });
}

Fonepay — Use Blocking Client

✅

Fonepay uses a URL redirect model and makes zero server-side HTTP calls. The blocking FonepayClient is 100% safe to use in reactive apps — it never touches the event loop. 

@Service
@RequiredArgsConstructor
public class FonepayService {

    // Blocking client — works in reactive apps (no HTTP calls)
    private final FonepayClient fonepayClient;

    public String buildRedirect(String orderId, double amountNPR) {
        String prn = "FP-" + orderId;
        // IMPORTANT: save prn to DB before returning!
        orderRepo.savePrn(orderId, prn);

        return fonepayClient.buildRedirectParams(
            FonepayPaymentRequest.builder()
                .prn(prn)
                .amount(amountNPR)      // NPR directly — not paisa
                .remarks1("Pro Plan")
                .build()
        ).redirectUrl();
    }

    // Wrap in Mono.fromCallable if needed in reactive pipeline
    public Mono<Boolean> verifyFonepay(
            FonepayCallbackResponse callback) {
        return Mono.fromCallable(() ->
            fonepayClient.verifyCallback(callback)
                         .isPaymentSuccessful());
    }
}

Retry

The reactive starter uses Reactor's Retry.backoff() — identical retry semantics to the blocking starter but fully non-blocking. 

nepalpay:
  khalti:
    retry:
      enabled: true           # disabled by default — opt-in
      max-attempts: 3         # retries after first failure
      initial-delay-ms: 500   # first retry after 500ms
      multiplier: 2.0         # 500 → 1000 → 2000ms
      max-delay-ms: 5000      # never wait more than 5s
Scenario Retried? Reason
5xx server errors ✅ Yes Gateway temporarily down
Network timeout / WebClientRequestException ✅ Yes Transient network issue
400 / 401 / 4xx ❌ No Bad request — retrying won't help
Signature failure ❌ No Security alert — must not retry
Gateway Methods with retry
Khalti initiatePayment(), lookupPayment(), refundPayment()
eSewa checkStatus() (inside verifyCallback())
ConnectIPS validateTransaction()
Fonepay N/A — No HTTP calls

Error Handling

All reactive clients emit typed gateway exceptions as Mono.error() signals — never thrown directly. Handle them with onErrorResume or onErrorMap. 

// Handle specific gateway exceptions
khaltiReactiveClient.initiatePayment(request)
    .onErrorResume(KhaltiException.class, ex -> {
        log.error("Khalti initiation failed | status={} | body={}",
            ex.httpStatus(), ex.responseBody());
        return Mono.error(
            new PaymentException("Payment gateway error", ex));
    });

// Handle all NepalPay exceptions in one place
khaltiReactiveClient.lookupPayment(pidx)
    .onErrorResume(NepalPayException.class, ex ->
        Mono.just(fallbackResponse()));

Exception hierarchy

Exception When emitted Fields
KhaltiException Any Khalti API failure httpStatus(), responseBody()
EsewaException eSewa API failure, signature mismatch httpStatus(), responseBody()
ConnectIpsException ConnectIPS API failure, RSA error httpStatus(), responseBody()
FonepayException Signature mismatch, invalid PRN, config error —
NepalPayException Base — catch all gateway errors —

Testing with StepVerifier

Use reactor-test's StepVerifier to assert on Mono responses. Combine with MockWebServer to mock HTTP responses.

Add test dependency

<dependency>
    <groupId>io.projectreactor</groupId>
    <artifactId>reactor-test</artifactId>
    <scope>test</scope>
</dependency>

Success assertion

@Test
void lookupPayment_completed_isSuccessful() {
    mockWebServer.enqueue(new MockResponse()
        .setResponseCode(200)
        .setHeader("Content-Type", "application/json")
        .setBody("""
            {"pidx":"bZQLD9","status":"Completed",
             "total_amount":10000}
            """));

    StepVerifier.create(
            khaltiReactiveClient.lookupPayment("bZQLD9"))
        .expectNextMatches(res ->
            res.isPaymentSuccessful()
                && res.totalAmount().equals(10000L))
        .verifyComplete();
}

Error signal assertion

@Test
void lookupPayment_nullPidx_emitsError() {
    StepVerifier.create(
            khaltiReactiveClient.lookupPayment(null))
        .expectErrorMatches(err ->
            err instanceof KhaltiException
                && err.getMessage().contains(
                    "pidx cannot be null"))
        .verify();

    // No HTTP call was made — validation rejected it
    assertThat(mockWebServer.getRequestCount()).isEqualTo(0);
}

Retry assertion

@Test
void lookupPayment_retryOnce_succeeds() {
    // First request fails with 500
    mockWebServer.enqueue(new MockResponse()
        .setResponseCode(500));
    // Second request succeeds
    mockWebServer.enqueue(new MockResponse()
        .setResponseCode(200)
        .setHeader("Content-Type", "application/json")
        .setBody("""
            {"pidx":"bZQLD9","status":"Completed",
             "total_amount":10000}
            """));

    StepVerifier.create(
            retryClient.lookupPayment("bZQLD9"))
        .expectNextMatches(KhaltiLookupResponse::isPaymentSuccessful)
        .verifyComplete();

    // Verify retry happened — 2 total requests
    assertThat(mockWebServer.getRequestCount()).isEqualTo(2);
}
