Security Best Practices

Payment integrations require extra security care. NepalPay enforces safe patterns by design — but you must follow these rules too.

✅ Always Do

1. Always verify server-side after Khalti redirect

// ✅ CORRECT
KhaltiLookupResponse lookup = khaltiClient.lookupPayment(pidx);
if (lookup.isPaymentSuccessful()) {
    markOrderAsPaid();
}

2. Always use verifyCallback() for eSewa

// ✅ CORRECT — does all 3 checks automatically
EsewaVerificationResult result = esewaClient.verifyCallback(data);
if (result.isPaymentSuccessful()) {
    markOrderAsPaid();
}

3. Verify amount matches your order

// ✅ Prevent amount-tampering attacks
if (!lookup.isAmountValid(expectedAmountPaisa)) {
    throw new SecurityException("Amount mismatch!");
}

4. Store pidx/transactionUuid before redirecting

// ✅ Save BEFORE redirecting user
orderRepo.savePidx(orderId, response.pidx());
return response.paymentUrl(); // then redirect

5. Use environment variables for secrets

nepalpay:
  khalti:
    secret-key: ${KHALTI_SECRET_KEY}  # ✅ from env var

❌ Never Do

❌ NEVER trust redirect parameters alone 
// ❌ WRONG — redirect can be faked by anyone
if (request.getParam("status").equals("Completed")) {
    markOrderAsPaid(); // VULNERABILITY!
}
❌ NEVER hardcode secrets in code or yml 
// ❌ WRONG — exposed in git history forever
secret-key: live_secret_key_abc123xyz
❌ NEVER put secrets in frontend code Secret keys must ONLY exist on your backend server. Never in Angular, React, or any client-side code.
